# File & Identifier Naming Conventions

The bootstrap prompts depend on consistent naming. This page is the authoritative list.

## File names

| Artifact | Convention | Example |
|---|---|---|
| Top-level project briefing | `CLAUDE.md` (literal, uppercase) | `CLAUDE.md` |
| Project info template input | `project-info.md` | `project-info.md` |

## Slug rules

Used for: epic-slug, decision-slug, task-slug, skill-slug, command-slug, role.

1. Lowercase ASCII only.
2. Separate words with `-` (kebab-case). Never `_`, never camelCase, never spaces.
3. Numbers OK (`v2`, `kafka-9092`).
4. â‰¤ 60 characters total. Truncate at the nearest word boundary.
5. Collapse consecutive hyphens (`add--feature` â†’ `add-feature`).
6. Trim leading/trailing hyphens.
7. No file extensions inside the slug.

Examples:

| Source | Slug |
|---|---|
| "Implement P2P transfer endpoint with idempotency" | `implement-p2p-transfer-endpoint-with-idempotency` |
| "Outbox vs publish-after-commit" | `outbox-or-publish-after-commit` |
| "Pessimistic locking on balance updates" | `pessimistic-locking-balance-updates` |
| "Add user â†’ wallet sync flow" | `add-user-wallet-sync-flow` |

## ADR numbering

- Four-digit zero-padded: `0001`, `0002`, â€¦
- **Never reuse a number**, even if the ADR is rejected or superseded. Superseded ADRs keep their number; the new ADR gets the next free one.
- Numbering is repo-wide, not per-topic.

## Migration numbering

(If `docs/database/migrations.md` declares Flyway-style versioning.)

- Format: `V<n>__<short_snake_case>.sql`
- Sequential, never reused.
- Single underscore in `V<n>` and double underscore separator.
- Forward-only â€” never edit a committed migration; write a compensating one.

Example: `V12__add_idempotency_keys_table.sql`.

## Rule section anchors

Rule files use numbered headings. The number is a **stable anchor** â€” once a section is numbered, its number doesn't change, because other files cite it.

- `## 1. Project structure`
- `### 1.1 Cross-feature rules`
- `### 1.2 Layer responsibilities`
- `## 2. Routing & Controllers`
- `### 2.1 Resource class layout`

**To insert a new section** without renumbering: append at the end of the file, or insert with a letter suffix (`### 2.3a`). Renumbering breaks every citation in skills, agents, commands, and docs â€” it must be done via the validation pass (step 7) with a global rename.

## Agent / Skill name conventions

- Agent name: `<role>-<noun>`, e.g. `backend-developer`, `frontend-developer`, `security-reviewer`. One name per agent file; the YAML frontmatter `name:` value must equal the filename (without `.md`).
- Skill name: `<area>-<verb>-<object>`, e.g. `backend-create-rest-api`, `frontend-implement-ui-component`, `code-review`, `backend-verify`. The folder name must equal the YAML frontmatter `name:`.
- Command name: `<verb>-<object>`, e.g. `make-plan`, `implement-plan`, `start-investigation`. The filename without `.md` equals the slash command the user types (`/make-plan`).

## Java/TypeScript package & module identifiers

(Only relevant when these guidelines generate code patterns inside rules.)

| Identifier | Convention | Example |
|---|---|---|
| Java package | reverse-DNS, lowercase, dots | `com.example.wallet.api` |
| Java class | PascalCase, no abbreviations | `TransferResource`, `WalletRepository` |
| Java constant | UPPER_SNAKE_CASE | `IDEMPOTENCY_KEY_HEADER` |
| TypeScript file | kebab-case | `transfer-form.component.ts` |
| TypeScript class | PascalCase | `TransferFormComponent` |
| TypeScript constant | UPPER_SNAKE_CASE | `MAX_PAGE_SIZE` |
| TypeScript object literal const | `as const`, camelCase keys | `API = { TRANSFERS: '/transfers' } as const` |

These mirror the canonical conventions in most ecosystems; rules generated by the bootstrap will use them unless `project-info.md Â§13` overrides.

## Environment variable conventions

- Uppercase, `SNAKE_CASE`.
- Namespace by stack component or by app: `KAFKA_BOOTSTRAP_SERVERS`, `APP_FRAUD_VOLUME_THRESHOLD`.
- Avoid double underscores unless the framework requires it (e.g. Microprofile `quarkus.foo.bar` â†’ `QUARKUS_FOO_BAR`).
- Never include credentials in the variable *name* â€” `DB_PASSWORD` is the secret value, not `DB_PROD_PASSWORD_REAL_FINAL`.

## Branch and PR conventions

(These map to `project-info.md Â§12`.)

- Branch prefix: `feature/`, `fix/`, `chore/`, `docs/`, `refactor/`. One word, single slash.
- Branch slug: derived from the task title via the slug rules above.
- Example: `feature/implement-p2p-transfer-endpoint`.
- PR title: â‰¤ 70 chars, imperative, no trailing period. ("Add P2P transfer endpoint with idempotency", not "Added the P2P transfer endpoint.").

## When in doubt

- Match the most-similar existing file in the repo. Consistency beats theory.
- If you're inventing a new artifact type, mirror the closest existing convention from this page and add a row when the bootstrap is next updated.
