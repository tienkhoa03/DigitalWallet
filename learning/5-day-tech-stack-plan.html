5 · Lý thuyết nền tảng xuyên suốt (học rải theo ngày)

Đây là các chủ đề “không gắn 1 stack” nhưng quyết định bạn code có “sạch” không. Mỗi mục được nhúng vào ngày tương ứng, đây là bản đồ tổng để bạn không bỏ sót.

  • Java core (Ngày 1): record, sealed class/interface, pattern matching (switch + record pattern), text block, var, Optional, Stream, generics, Comparable/equality, BigDecimal cho tiền.
  • Concurrency cơ bản (Ngày 1): thread vs virtual thread (Project Loom), vì sao Java 21, blocking I/O trên virtual thread.
  • OOP & Clean Code (Ngày 1–2): encapsulation, composition over inheritance, đặt tên, hàm nhỏ, tránh primitive obsession (→ value object).
  • SOLID (Ngày 2): SRP, OCP, LSP, ISP, DIP — đặc biệt DIP là “xương sống” của hexagonal.
  • DDD tactical (Ngày 2): entity, value object, aggregate, repository, domain service vs application service, ubiquitous language (repo đã có glossary: Account, Wallet, Transfer…).
  • Kiến trúc (Ngày 2): layered → onion → hexagonal; “dependency rule” (domain không phụ thuộc ai).
  • Distributed systems & resilience (Ngày 3): timeout, retry, circuit breaker, bulkhead, fallback, idempotency, at-least-once, health liveness/readiness, metrics RED/USE.
  • 12-factor & observability (Ngày 3): config qua env, logs là stream, backing services, 3 trụ observability (metrics/logs/traces).
  • Security (Ngày 4): AuthN vs AuthZ, RBAC, OAuth2 roles & grants, OIDC, cấu trúc JWT, JWKS, validate token, password hashing (repo dùng Argon2).
  • Frontend & IaC (Ngày 5): SPA, component model, reactivity (ref/reactive), declarative UI; IaC: desired state, plan/apply, state file, provider/resource.

Thực hành 1.A Chạy stack & đọc slice account end-to-end
  1. Bật hạ tầng: docker compose -f backend/docker-compose.yml up -d
  2. Chạy dev mode: cd backend && ./mvnw quarkus:dev (thử sửa 1 chuỗi → live reload).
  3. Gọi thử API: POST /accounts (signup) rồi POST /auth/login để lấy JWT. Dùng curl/Postman/HTTPie.
  4. Đọc theo thứ tự dòng chảy & ghi chú mỗi class thuộc “lớp” nào: AccountResource → AccountService → AccountRepository → AccountEntity → V1 migration.
  5. Đọc thêm DomainException (sealed) + DomainExceptionMapper để hiểu cách lỗi → HTTP envelope.

Output: một bảng tự ghi “concept Quarkus → file trong repo” (CDI bean? resource? repository? transaction boundary ở đâu?).

Thực hành 1.B Tự thêm endpoint GET /accounts/me

Đây là bài “đi hết slice” đầu tiên, dùng JWT có sẵn (extension smallrye-jwt đã bật) để lấy danh tính.

  1. Tạo record AccountProfileResponse trong account/dto/ (id, email, baseCurrency, role) — KHÔNG lộ password_hash (security.md §7).
  2. Trong AccountResource: thêm method @GET @Path("/me") @RolesAllowed("USER"), inject JsonWebToken jwt, lấy jwt.getSubject() làm accountId.
  3. Trong AccountService: thêm getProfile(UUID id) trả DTO; dùng AccountRepository.findByIdOptional, trả Optional → map sang DTO, không có thì ném DomainException phù hợp.
  4. Giữ constructor injection; @Transactional(SUPPORTS) hoặc bỏ annotation vì chỉ đọc (backend_coding.md §3).
  5. Test thủ công: gọi GET /accounts/me kèm header Authorization: Bearer <token>; thử không token → kỳ vọng 401.
Nhớ

Ownership = lấy id từ token, không nhận id từ client. Đây chính là rule “ownership check ở service layer” (security.md §3).

Thực hành 1.C Unit test cho service mới (JUnit 5 + Mockito)
  1. Tạo AccountServiceTest: @Mock AccountRepository, @InjectMocks AccountService.
  2. Đặt tên theo chuẩn repo: getProfile_givenExistingId_returnsProfile và getProfile_givenMissingId_throwsDomainException (testing.md §2.3).
  3. Assert bằng AssertJ: assertThat(...); với exception dùng assertThatThrownBy(...) và assert cả errorKey (testing.md §2.5).
  4. Chạy: ./mvnw -pl backend test -Dtest=AccountServiceTest (hoặc skill /backend-create-unit-test để sinh khung).

Thực hành 2.A Refactor account sang ports & adapters

Cấu trúc đích (đặt trong com/digitalwallet/account/):

account/
├── domain/                      # KHÔNG import jakarta.persistence / quarkus
│   ├── Account.java             # domain model (record/class thuần)
│   ├── AccountId.java  Email.java  BaseCurrency.java   # value objects
│   └── AccountRole.java
├── application/
│   ├── port/in/                 # inbound ports (use cases)
│   │   ├── RegisterAccountUseCase.java
│   │   └── AuthenticateUseCase.java
│   ├── port/out/                # outbound ports (interfaces)
│   │   ├── LoadAccountPort.java   SaveAccountPort.java
│   │   ├── PasswordHasherPort.java  TokenIssuerPort.java
│   └── service/
│       └── AccountApplicationService.java   # implements in-ports, dùng out-ports
└── adapter/
    ├── in/web/                  # AccountResource, AuthResource (REST → command)
    └── out/
        ├── persistence/         # AccountPersistenceAdapter + AccountEntity + Panache repo
        └── security/            # adapter bọc JwtIssuer + Argon2Hasher
  1. Tạo domain/Account thuần + value objects (Email, BaseCurrency) — validate bất biến trong constructor (primitive obsession → value object).
  2. Định nghĩa out-ports: LoadAccountPort.findByEmail(...), SaveAccountPort.save(Account), PasswordHasherPort, TokenIssuerPort.
  3. Chuyển logic từ AccountService/AuthService vào AccountApplicationService, chỉ gọi out-ports.
  4. Viết AccountPersistenceAdapter implement port out: bọc AccountEntity + Panache repo, map entity ↔ domain (2 hàm toDomain/toEntity).
  5. Viết SecurityAdapter implement TokenIssuerPort/PasswordHasherPort bằng JwtIssuer + Argon2Hasher có sẵn.
  6. Resource (adapter/in/web) chỉ map DTO ↔ command rồi gọi in-port. Giữ nguyên path & response để không phá test.
Trade-off cần ghi chú

Tách domain khỏi JPA tốn thêm lớp mapping entity↔domain. Với Panache, nhiều team chấp nhận entity “lai”. Hãy ghi lại cảm nhận: lợi (test domain không cần DB) vs chi phí (boilerplate). Đây đúng là tinh thần “học để hiểu trade-off”.

Thực hành 2.B Bảo vệ dependency rule bằng ArchUnit (tuỳ chọn)
  • ArchUnit: archunit.org
  1. Thêm dependency test com.tngtech.archunit:archunit-junit5.
  2. Viết rule: package ..account.domain.. KHÔNG được “depend on” jakarta.persistence.., io.quarkus.., ..adapter...
  3. Chạy ./mvnw -pl backend test — toàn bộ test cũ + ArchUnit phải xanh.
⚠ Lệch stack

Cấu trúc này mâu thuẫn rule “feature + layered” (backend_coding.md §1). Giữ trong nhánh learning/*. Nếu muốn áp dụng cho team → viết ADR mới supersede docs/architecture/README.md §3 (theo quy trình upgrade-policy.md §5).

  • Toàn bộ test cũ vẫn xanh (hành vi không đổi).
  • Có sơ đồ entity↔domain + ghi chú trade-off.
  • (tuỳ chọn) ArchUnit chặn phụ thuộc ngược.

Lý thuyết 3.1 12-factor + resilience patterns

  • Khái niệm: timeout, retry (+ exponential backoff), bulkhead, fallback, idempotency, at-least-once.

Vì sao: NFR8 của repo yêu cầu circuit breaker bọc LLM; NFR5 yêu cầu cô lập latency. Hôm nay bạn làm “bản nháp” của chính các pattern đó.

Vì sao: MicroProfile = bộ spec cho microservice Java; Quarkus hiện thực qua SmallRye. Repo đã có MP Config + JWT + Micrometer; hôm nay thêm Health/Fault-Tolerance/OpenAPI để “đủ bộ”.

Thực hành 3.A Dựng wallet MVP slice (viết theo hexagonal của Ngày 2)
  1. Thêm extensions: ./mvnw quarkus:add-extension -Dextensions="smallrye-health,smallrye-fault-tolerance,smallrye-openapi" (micrometer-prometheus đã có).
  2. Flyway V2__create_wallet_table.sql: cột id uuid pk, account_id uuid, currency_code varchar(3), label varchar(64), balance numeric(19,4), created_at timestamptz; UNIQUE (account_id, label) (entity + migration cùng PR — backend_coding.md §13).
  3. Domain + ports: Wallet, OpenWalletUseCase, DepositUseCase, ListWalletsUseCase; out-port WalletPersistencePort.
  4. Endpoints trong wallet/adapter/in/web/WalletResource:
    • POST /wallets — mở ví (OpenWalletRequest{currencyCode,label}, @Valid, currency ISO 4217 dùng @CurrencyCode có sẵn).
    • POST /wallets/{walletId}/deposits — nhận header Idempotency-Key (chỉ nhận, chưa enforce replay — NFR3 ngoài scope), body amount > 0 (BigDecimal).
    • GET /wallets — liệt kê ví của chủ token.
  5. Ownership ở service: account lấy từ JWT phải sở hữu ví (security.md §3). Deposit chỉ @Transactional đơn giản (KHÔNG Redis lock — descoped).
⚠ Đây là deposit “đồ chơi”

Thiếu NFR1/2/3 nên KHÔNG dùng cho production. Mục tiêu là học REST + Panache + transaction + MicroProfile, không phải money-path thật.

Thực hành 3.B Gắn MicroProfile lên slice
  1. Health (R2): thay HealthResource tự viết bằng @Readiness check ping DB; xem /q/health/live & /q/health/ready.
  2. Metrics: đặt @Timed("wallet.deposit") trên use case deposit; mở /q/metrics tìm timer; nạp vài lần và xem count tăng.
  3. OpenAPI/Swagger (R3): chú thích @Operation, @APIResponse trên resource; mở /q/swagger-ui và gọi thử toàn bộ wallet API ngay trên UI.
Thực hành 3.C Demo Circuit Breaker (mô phỏng NFR8)
  1. Tạo out-port giả FxRatePort + adapter FlakyFxRateAdapter cố tình throw/chậm ngẫu nhiên.
  2. Bọc @CircuitBreaker(requestVolumeThreshold=4, failureRatio=0.5) + @Retry(maxRetries=2) + @Fallback(fallbackMethod="defaultRate").
  3. Gọi liên tục → quan sát breaker “open”, fallback trả giá trị mặc định thay vì lỗi.

Vì sao: đây chính xác là khuôn mẫu repo cần cho advisor/LLM (NFR8). Học ở đây để sau ráp vào module advisor thật.

Thực hành 3.D Test (giữ coverage service ≥ 80% — NFR4)
  • Unit: test DepositUseCase/OpenWalletUseCase với out-port mock; boundary amount tại 0, 0.0001, số âm (testing.md §2.6); dùng BigDecimal.compareTo.
  • Integration: Testcontainers Postgres cho persistence adapter (Flyway chạy lúc start — KHÔNG dùng H2).
  • Fault-tolerance: test ép breaker open → assert fallback được gọi.
  • Chạy gate: cd backend && ./mvnw verify (JaCoCo 80% trên service/**).

Definition of Done — Ngày 3

  • 3 endpoint wallet hoạt động, thử được trên /q/swagger-ui.
  • /q/health/ready phản ánh trạng thái DB; timer deposit hiện ở /q/metrics.
  • Circuit breaker mở được & fallback chạy (có test chứng minh).
  • ./mvnw verify xanh, coverage service ≥ 80%.

Tự kiểm: liveness khác readiness ở đâu? Khi nào nên retry vs khi nào mở breaker? Vì sao metrics phải là pull (/q/metrics) cho Prometheus?

Keycloak — OAuth2/OIDC bảo vệ API wallet

Thực hành5.0hDựng Keycloak + realm (1.5h) · Quarkus OIDC + bảo vệ wallet (2h) · lấy token & gọi API (1h) · auth tests (0.5h)
Tổng kết0.5hSơ đồ luồng authorization-code + bearer token
Lý thuyết 4.1 OAuth2 & OIDC & JWT
  • OAuth 2 Simplified (Aaron Parecki): aaronparecki.com/oauth-2-simplified
  • How OIDC works: openid.net/developers/how-connect-works
  • RFC 6749 (tra cứu grant types): datatracker.ietf.org/doc/html/rfc6749

Nắm chốt: 4 vai (resource owner, client, authorization server, resource server); grant authorization code + PKCE (cho SPA Vue) vs client credentials (server-server); OIDC thêm ID token bên cạnh access token; access token là JWT verify bằng JWKS của Keycloak.

Lý thuyết 4.2 Keycloak concepts + Quarkus OIDC
  • Keycloak getting started (Docker): keycloak.org/getting-started/getting-started-docker
  • Quarkus — OIDC bảo vệ service: quarkus.io/guides/security-openid-connect
  • Quarkus — Bearer token auth: security-oidc-bearer-token-authentication
  • Quarkus — Keycloak authorization: security-keycloak-authorization

Nắm chốt: realm (không gian người dùng) → client (ứng dụng) → role (realm vs client) → user. Quarkus OIDC kiểu service chỉ verify bearer token; map realm role vào @RolesAllowed.

Thực hành 4.A Dựng Keycloak + realm digitalwallet
  1. Thêm service keycloak (dev mode) vào backend/docker-compose.yml (tạm để học; Ngày 5 chuyển sang Terraform). Map cổng 8081:8080, admin admin/admin.
  2. Vào console, tạo realm digitalwallet.
  3. Tạo role realm: USER, ADMIN.
  4. Tạo client digitalwallet-backend (confidential, dùng để verify) và digitalwallet-frontend (public, Standard Flow bật — dành cho Vue Ngày 5; redirect URI http://localhost:5173/*).
  5. Tạo user test (vd alice), đặt password, gán role USER.
  6. Export realm ra infra/keycloak/realm-export.json để tái lập (và import bằng Terraform sau).
Thực hành 4.B Quarkus OIDC bảo vệ wallet (R5, theo profile)
  1. Thêm extension: ./mvnw quarkus:add-extension -Dextensions="oidc".
  2. Cấu hình profile %keycloak trong application.properties: 
    %keycloak.quarkus.oidc.auth-server-url=http://localhost:8081/realms/digitalwallet
%keycloak.quarkus.oidc.client-id=digitalwallet-backend
%keycloak.quarkus.oidc.application-type=service
%keycloak.quarkus.oidc.roles.role-claim-path=realm_access/roles
  3. Giữ @RolesAllowed("USER") trên wallet resource — giờ role đến từ token Keycloak.
  4. Chạy: ./mvnw quarkus:dev -Dquarkus.profile=keycloak. Path ES256 cũ vẫn còn ở profile mặc định (không xoá).
⚠ Lệch ADR 0001

ADR 0001 (ES256 tự ký) đang “Accepted”. Đây là spike song song theo profile; muốn thay thật phải viết ADR supersede 0001 (security.md §2, upgrade-policy.md §5).

Thực hành 4.C Lấy token & kiểm thử bảo mật
  1. Lấy access token qua token endpoint (password grant cho mục đích test): 
    curl -X POST \
  http://localhost:8081/realms/digitalwallet/protocol/openid-connect/token \
  -d grant_type=password -d client_id=digitalwallet-frontend \
  -d username=alice -d password=<pw>
  2. Gọi GET /wallets kèm Authorization: Bearer <access_token> → 200.
  3. Không token → 401; user thiếu role USER → 403 (security.md §11 ma trận test).
  4. Viết 2 test (RestAssured/QuarkusTest): unauthenticated → 401, wrong-role → 403.

Definition of Done — Ngày 4

  • Keycloak chạy; realm digitalwallet có client + role + user, đã export JSON.
  • Backend (profile keycloak) verify được token Keycloak; map role → RBAC.
  • Lấy được access token và gọi /wallets thành công.
  • Test 401/403 xanh; path ES256 cũ vẫn nguyên.
  • Vẽ được luồng auth-code + PKCE (chuẩn bị cho Vue).

Tự kiểm: ID token khác access token ở đâu? Vì sao SPA dùng auth-code+PKCE chứ không password grant? JWKS giúp verify token không cần gọi lại Keycloak mỗi request thế nào?

Thực hành 5.A Scaffold app Vue + login Keycloak
  1. Tạo app tách biệt: pnpm create vue@latest frontend-vue (bật TypeScript, Router, Pinia). KHÔNG đụng frontend/ (React).
  2. Cài keycloak-js; khởi tạo với client public digitalwallet-frontend (auth-code + PKCE) trỏ realm digitalwallet.
  3. Pinia auth.store: giữ token in-memory (không để localStorage — security.md §6), action login()/logout().
  4. Route guard requireAuth: chưa đăng nhập → redirect Keycloak.
Thực hành 5.B Màn hình ví (gọi API đã bảo vệ)
  1. Pinia wallet.store: action fetchWallets() gọi GET /wallets kèm Authorization: Bearer từ auth store.
  2. Trang WalletList.vue: hiện label + currency_code + balance cùng nhau (không định danh ví bằng currency — mirror rule frontend repo). Format tiền bằng Intl.NumberFormat, KHÔNG toFixed trên số (giữ chính xác numeric(19,4)).
  3. Form OpenWallet.vue + Deposit.vue: gọi POST /wallets & POST /wallets/{id}/deposits; deposit sinh Idempotency-Key (UUID) lúc mount, dùng lại khi resubmit.
  4. Cấu hình proxy Vite /api → backend (cổng 8080) hoặc bật CORS allow-list cho http://localhost:5173 (security.md §5).

Phần B — Terraform

Lý thuyết 5.2 IaC & Terraform cơ bản

Nắm chốt: declarative/desired-state, init/plan/apply/destroy, state file, provider/resource/variable/output/module, tính idempotent của apply.

Thực hành 5.C Terraform dựng hạ tầng local (R6)

Tạo infra/terraform/ tái lập đúng stack đang chạy tay trong backend/docker-compose.yml, nhưng bằng resource Terraform.

  1. main.tf: provider kreuzwerker/docker; network dw-net; docker_image + docker_container cho Postgres 16, Redis 7, Kafka (KRaft), Keycloak.
  2. variables.tf: cổng, mật khẩu (truyền qua TF_VAR_*, KHÔNG hardcode secret — security.md §1); outputs.tf: tên container + cổng map.
  3. Mount infra/keycloak/realm-export.json để import realm tự động khi container Keycloak start.
  4. Chạy: terraform init → terraform plan → terraform apply; kiểm tra container; rồi terraform destroy để thấy vòng đời IaC.
⚠ Lệch §4.6

Repo mandate Docker Compose. Terraform ở đây sống dưới infra/terraform/ như spike song song, không thay compose của team trừ khi có ADR.

Thực hành 5.D Demo End-to-End
  1. terraform apply → hạ tầng (gồm Keycloak) lên.
  2. Chạy backend ./mvnw quarkus:dev -Dquarkus.profile=keycloak.
  3. pnpm --dir frontend-vue dev → mở http://localhost:5173 → login qua Keycloak → thấy danh sách ví → nạp tiền → balance cập nhật.
  4. Quay video/screenshot 60 giây làm “bằng chứng học”.

Definition of Done — Ngày 5

  • Vue app login qua Keycloak, token in-memory.
  • List ví + open + deposit chạy thật với backend đã bảo vệ.
  • Tiền hiển thị bằng Intl.NumberFormat, ví hiện label+currency.
  • terraform apply dựng đủ Postgres/Redis/Kafka/Keycloak; destroy sạch.
  • Demo E2E thành công, có bản ghi lại.

Tự kiểm: Pinia store khác Redux ở điểm nào? Vì sao token để in-memory an toàn hơn localStorage? terraform plan giúp gì trước khi apply?
