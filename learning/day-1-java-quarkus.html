Lý thuyết · Phần 1/7Quarkus là gì & vì sao nó nhanh

Quarkus là một framework Java “container-first”: thay vì làm phần lớn công việc lúc chạy (như Spring Boot truyền thống), nó dồn việc sang lúc build.

Một app Spring Boot khi khởi động phải: quét classpath, đọc annotation bằng reflection, dựng cây dependency injection, cấu hình ORM… Tất cả diễn ra lúc khởi động → chậm, tốn RAM. Quarkus làm hầu hết những việc đó lúc build (build-time augmentation) và “đóng băng” kết quả vào bytecode. Khi chạy, app gần như chỉ việc bật lên.

Hệ quả tích cực

  • Khởi động mili-giây, RAM thấp → hợp container/serverless.
  • Biên dịch được sang native image (GraalVM) — repo có profile native trong pom.xml.
  • Lỗi cấu hình lộ ra lúc build chứ không phải lúc deploy.

Đánh đổi cần biết

  • Một số thư viện dùng reflection “động” cần “extension” riêng để hoạt động.
  • Build lâu hơn chút (vì làm nhiều việc hơn lúc build).
  • Tư duy “extension” thay vì “thả jar bất kỳ” (xem upgrade-policy.md §3: ưu tiên extension Quarkus).

Dev mode & live reload — thứ bạn sẽ dùng cả ngày

./mvnw quarkus:dev bật chế độ phát triển: sửa file .java rồi gọi lại API là Quarkus tự biên dịch lại phần thay đổi (không cần restart). Có sẵn Dev UI tại http://localhost:8080/q/dev/ để xem bean, config, health…

Dev Services

Khi chạy test (hoặc dev mà chưa cấu hình DB), Quarkus có thể tự bật một container Postgres tạm bằng Testcontainers — gọi là Dev Services. Đó là lý do test tích hợp “tự có DB” mà bạn không phải khai báo gì.

✔ Tự kiểm

Lý thuyết · Phần 2/7CDI / ArC — tiêm phụ thuộc (Dependency Injection)

CDI (Contexts and Dependency Injection) là cách Quarkus “lắp ráp” các đối tượng cho bạn. Bạn khai báo một class là bean, CDI lo việc tạo và đưa nó vào nơi cần. Quarkus hiện thực CDI bằng engine tên ArC.

Scope — vòng đời của bean

AnnotationÝ nghĩaDùng cho
@ApplicationScoped1 instance dùng chung toàn app (lazy)Service, repository — mặc định nên dùng
@RequestScoped1 instance mỗi HTTP requestState theo request (hiếm khi tự viết)
@SingletonGiống ApplicationScoped nhưng tạo sớm, không proxyBean hạ tầng đơn giản

Constructor injection — luật bắt buộc của repo

Có 2 kiểu tiêm: qua field (@Inject trên thuộc tính) và qua constructor. Repo này cấm field injection (upgrade-policy.md §3, backend_coding.md §3) vì constructor injection giúp: (1) field final bất biến, (2) test dễ (truyền mock vào constructor), (3) lộ rõ phụ thuộc.

Ví dụ minh hoạ — đúng cách
@ApplicationScoped
public class AccountService {

    private final AccountRepository repository;
    private final Argon2Hasher hasher;

    // CDI tự gọi constructor này và truyền 2 bean vào
    public AccountService(AccountRepository repository, Argon2Hasher hasher) {
        this.repository = repository;
        this.hasher = hasher;
    }
}
Sai (đừng làm): @Inject AccountRepository repository; đặt trên field. Code review của repo sẽ chặn.

Lý thuyết · Phần 3/7Quarkus REST (JAX-RS) — tầng controller

Endpoint HTTP được khai báo bằng annotation JAX-RS (chuẩn jakarta.ws.rs.*). Extension trong repo tên quarkus-rest (chính là RESTEasy Reactive thế hệ mới).

AnnotationVai trò
@Path("/accounts")Đường dẫn gốc của resource / method
@GET @POST @PUT @PATCH @DELETEHTTP method
@PathParam @QueryParam @HeaderParamLấy tham số từ URL/header
@ValidKích hoạt Bean Validation cho body
@RolesAllowed("USER")Chặn theo vai trò (RBAC) — chuẩn jakarta.annotation.security
Mẫu giống AccountResource trong repo
@Path("/accounts")
@Produces(MediaType.APPLICATION_JSON)
@Consumes(MediaType.APPLICATION_JSON)
public class AccountResource {

    private final AccountService service;
    public AccountResource(AccountService service) { this.service = service; }

    @POST
    public RestResponse<CreateAccountResponse> signup(@Valid CreateAccountRequest req) {
        return RestResponse.status(Status.CREATED, service.register(req));
    }
}
Luật repo

Resource chỉ trả DTO (record), không bao giờ trả thẳng JPA entity (backend_coding.md §6). Và không đặt @Transactional ở resource — nó thuộc service.

Lý thuyết · Phần 4/7Hibernate ORM + Panache — tầng truy cập dữ liệu

Panache là lớp “đường ngọt” trên Hibernate giúp viết DB code ngắn gọn. Có 2 phong cách; repo này chọn Repository pattern (không phải Active Record) để entity sạch, dễ mock khi test.

Entity

Class ánh xạ tới bảng. Dùng @Entity, @Table, @Id, @Column. Tiền là BigDecimal ↔ numeric(19,4); thời gian là Instant/OffsetDateTime ↔ timestamptz.

Repository

Implement PanacheRepositoryBase<Entity, UUID>. Có sẵn findById, persist, list, find(...). Trả Optional cho tra cứu 1 dòng (repo cấm trả null — backend_coding.md §5).

Ví dụ minh hoạ — repository & truy vấn có tham số ràng buộc
@ApplicationScoped
public class AccountRepository implements PanacheRepositoryBase<AccountEntity, UUID> {

    public Optional<AccountEntity> findByEmail(String email) {
        // dùng named parameter (:email) — KHÔNG nối chuỗi (chống injection)
        return find("email", email).firstResultOptional();
    }
}

Transaction đặt ở đâu?

@Transactional đặt ở method của service — không phải resource, không phải repository. Method vừa đọc vừa ghi ledger thì bắt buộc @Transactional; method chỉ đọc thì dùng @Transactional(TxType.SUPPORTS) hoặc bỏ hẳn.

REQUEST │ ▼ ┌──────────────┐ trả DTO ┌──────────────┐ @Transactional ┌──────────────┐ │ Resource │ ───────────► │ Service │ ───────────────► │ Repository │ ──► Postgres │ (JAX-RS) │ │ (nghiệp vụ) │ │ (Panache) │ └──────────────┘ └──────────────┘ └──────────────┘ @RolesAllowed ownership check named params

Lý thuyết · Phần 5/7Flyway (migration) & Config

Flyway — schema bằng SQL có phiên bản

Schema DB không được Hibernate tự sinh (repo đặt quarkus.hibernate-orm.database.generation=none). Mọi thay đổi schema là file SQL versioned: V<số>__<mô_tả>.sql trong backend/src/main/resources/db/migration/. Repo đã có V1__create_account_table.sql. Quy tắc vàng: entity và migration đi cùng một PR (backend_coding.md §13).

Config — MicroProfile Config

Giá trị cấu hình nằm trong application.properties, đọc bằng @ConfigProperty hoặc @ConfigMapping (cấm gọi thẳng System.getenv). Có 3 profile: dev, test, prod; ghi đè bằng tiền tố %dev., %test.. Secret (mật khẩu DB, khoá JWT) không có default trong file — phải đến từ biến môi trường (security.md §1).

Ví dụ minh hoạ — đọc config
public class SomeService {
    private final int pageSizeCap;
    public SomeService(@ConfigProperty(name = "app.page-size-cap", defaultValue = "100") int cap) {
        this.pageSizeCap = cap;
    }
}

Lý thuyết · Phần 6/7Java 21 — 6 viên gạch bạn gặp khắp repo

1) record — DTO bất biến

Record sinh sẵn constructor, getter, equals/hashCode/toString. Repo dùng record cho mọi DTO. Có thể thêm compact constructor để validate.

Ví dụ
public record MoneyAmount(BigDecimal value, String currencyCode) {
    public MoneyAmount {                       // compact constructor
        if (value.signum() <= 0)
            throw new IllegalArgumentException("amount must be > 0");
    }
}

2) sealed interface — tập con đóng

Cho phép liệt kê chính xác các class con. Repo dùng cho DomainException permits ValidationException, ConflictException, … để bộ switch xử lý lỗi không bao giờ “quên case”.

3) Pattern matching switch

Ví dụ — map exception → HTTP status gọn gàng
int status = switch (ex) {
    case ValidationException v   -> 400;
    case AuthForbiddenException f -> 403;
    case ConflictException c      -> 409;
    default                       -> 422;
};

4) Optional — không trả null

repository.findByIdOptional(id).orElseThrow(() -> new SomeException(...)). Tránh NullPointerException và làm rõ “có thể không tồn tại”.

5) BigDecimal cho tiền — TUYỆT ĐỐI không double

Bẫy kinh điển: 0.1 + 0.2 == 0.30000000000000004 với double. Tiền phải dùng BigDecimal. So sánh giá trị bằng compareTo chứ không equals — vì new BigDecimal("1.0").equals(new BigDecimal("1.00")) trả false (khác scale) nhưng compareTo trả 0.

6) Text block & var

Text block """...""" cho SQL/JSON nhiều dòng (đỡ nối + "\n"). var cho biến local khi kiểu đã rõ ở vế phải — nhưng giữ kiểu tường minh cho BigDecimal, UUID, Instant để dễ đọc.

Lý thuyết · Phần 7/7Virtual threads (Project Loom) — chỉ cần hiểu khái niệm

Thread truyền thống của OS “nặng” (~1MB stack) nên không thể tạo hàng triệu. Virtual thread (Java 21) siêu nhẹ, JVM quản lý: khi một virtual thread block chờ I/O (gọi DB, gọi Kafka), nó “nhường” OS thread cho việc khác. Kết quả: viết code blocking quen thuộc (dễ đọc) nhưng vẫn scale như code async.

Trong Quarkus, đánh dấu @RunOnVirtualThread trên endpoint/consumer để chạy trên virtual thread. Repo chọn Java 21 chính vì điều này (cho Kafka consumer & HTTP — CLAUDE.md). Hôm nay chưa cần dùng, chỉ cần hiểu “vì sao Java 21”.

✔ Tự kiểm cuối phần lý thuyết

  • Record khác class thường chỗ nào? (bất biến + sinh sẵn equals/hashCode/getter)
  • Vì sao tiền không dùng double? So sánh BigDecimal bằng gì?
  • @Transactional đặt ở tầng nào?

Thực hành · 3/4Viết endpoint GET /accounts/me (≈ 90 phút)

Mục tiêu: lấy danh tính người gọi từ JWT, trả profile (ẩn password_hash). Đây là bài “đi hết slice” đầu tiên của bạn.

3.1 — Tạo DTO response (record)

Tạo file account/dto/AccountProfileResponse.java:

Gõ tương tự vào repo
package com.digitalwallet.account.dto;

import java.util.UUID;

public record AccountProfileResponse(
        UUID id,
        String email,
        String baseCurrency,
        String role) { }

3.2 — Thêm method service (chỉ đọc)

Trong AccountService, thêm:

Gõ tương tự vào repo
@Transactional(Transactional.TxType.SUPPORTS)   // chỉ đọc
public AccountProfileResponse getProfile(UUID accountId) {
    return repository.findByIdOptional(accountId)
        .map(a -> new AccountProfileResponse(
                a.id, a.email, a.baseCurrency, a.role.name()))
        .orElseThrow(() -> new AuthForbiddenException(
                "auth.forbidden", "Account not found"));
}

Tên field/exception chỉ là minh hoạ — chỉnh theo đúng class thật trong repo (xem AccountEntity và package shared/exception).

3.3 — Thêm method resource lấy danh tính từ JWT

Extension quarkus-smallrye-jwt đã bật sẵn, nên có thể inject JsonWebToken. jwt.getSubject() chính là accountId đã được ký lúc login.

Gõ tương tự vào AccountResource
import org.eclipse.microprofile.jwt.JsonWebToken;
import jakarta.annotation.security.RolesAllowed;

// inject qua constructor (KHÔNG field injection)
private final AccountService service;
private final JsonWebToken jwt;
public AccountResource(AccountService service, JsonWebToken jwt) {
    this.service = service; this.jwt = jwt;
}

@GET @Path("/me")
@RolesAllowed("USER")
public AccountProfileResponse me() {
    UUID accountId = UUID.fromString(jwt.getSubject());   # danh tính từ token, KHÔNG nhận từ client
    return service.getProfile(accountId);
}
Vì sao lấy id từ token?

Đây là ownership check (security.md §3): nếu nhận accountId từ client, user A có thể đọc profile user B. Lấy từ token (sub) thì không thể giả mạo.

3.4 — Thử nghiệm thủ công

Dùng token lấy ở Thực hành 1
TOKEN=...<dán token>...
curl -i http://localhost:8080/accounts/me -H "Authorization: Bearer $TOKEN"   # 200 + profile
curl -i http://localhost:8080/accounts/me                                     # 401 (không token)
✅ Checkpoint: có token → 200 và không thấy password_hash trong JSON; không token → 401.

Thực hành · 4/4Viết unit test (JUnit 5 + Mockito) (≈ 60 phút)

Test service mock repository — không cần DB. Đặt tên theo chuẩn repo và assert cả errorKey khi test exception.

backend/src/test/java/com/digitalwallet/account/service/AccountServiceProfileTest.java
@ExtendWith(MockitoExtension.class)
class AccountServiceProfileTest {

    @Mock  AccountRepository repository;
    @InjectMocks AccountService service;

    @Test
    void getProfile_givenExistingId_returnsProfile() {
        var id = UUID.randomUUID();
        var entity = new AccountEntity(/* ... id, email, USD, USER ... */);
        when(repository.findByIdOptional(id)).thenReturn(Optional.of(entity));

        var res = service.getProfile(id);

        assertThat(res.email()).isEqualTo("alice@example.com");
        assertThat(res.baseCurrency()).isEqualTo("USD");
    }

    @Test
    void getProfile_givenMissingId_throwsForbidden() {
        var id = UUID.randomUUID();
        when(repository.findByIdOptional(id)).thenReturn(Optional.empty());

        assertThatThrownBy(() -> service.getProfile(id))
            .isInstanceOf(AuthForbiddenException.class)
            .extracting("errorKey")            # bắt buộc assert errorKey
            .isEqualTo("auth.forbidden");
    }
}
Chạy test
cd backend
./mvnw test -Dtest=AccountServiceProfileTest
