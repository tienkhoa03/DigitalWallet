Lý thuyết · 2/4Resilience patterns — chống lỗi trong hệ phân tán

Khi app gọi service ngoài (DB, LLM, FX API), thứ ngoài đó sẽ chậm hoặc chết. 5 pattern dưới đây giúp lỗi cục bộ không kéo sập cả hệ thống.

PatternGiải quyếtKhi nào dùng
TimeoutCuộc gọi treo vô hạnMọi cuộc gọi I/O ra ngoài
Retry (+ backoff)Lỗi thoáng qua (mạng chớp tắt)Lỗi có khả năng tự khỏi; không retry lỗi logic (400/422)
Circuit BreakerService ngoài chết kéo dài → đừng gọi nữa, để nó hồiKhi retry chỉ làm tệ hơn
BulkheadMột phần chậm “ăn” hết thread của phần khácCô lập pool theo loại việc
FallbackCần một câu trả lời “tạm chấp nhận” khi lỗiTrả giá trị mặc định / cache cũ

Circuit breaker — 3 trạng thái (nhớ kỹ)

lỗi vượt ngưỡng hết thời gian chờ CLOSED ───────────────────► OPEN ───────────────────► HALF-OPEN ▲ (cho gọi bình thường) (chặn ngay, (cho gọi thử vài request) │ đếm tỉ lệ lỗi) trả fallback) │ └───────────────────────────────────────────────────────┘ thử thành công → đóng lại (CLOSED)

Liên hệ repo: NFR8 yêu cầu bọc circuit breaker quanh cuộc gọi LLM của advisor. Bài demo hôm nay là “bản nháp” của chính pattern đó.

Lý thuyết · 3/4Observability — health & metrics

Health: liveness vs readiness

  • Liveness = “app còn sống không?” Nếu fail → orchestrator restart container.
  • Readiness = “app sẵn sàng nhận traffic chưa?” (vd DB đã kết nối). Nếu fail → tạm ngừng route traffic, không restart.
  • Endpoint Quarkus: /q/health/live, /q/health/ready, /q/health.

Metrics: RED method

  • Rate — số request/giây.
  • Errors — tỉ lệ lỗi.
  • Duration — độ trễ (timer/histogram).
  • Repo dùng Micrometer + Prometheus (đã có extension). Prometheus kéo số liệu từ /q/metrics (pull-based).

Thực hành · 2/7Dựng wallet slice (theo hexagonal Ngày 2) (≈ 120 phút)

Tái dùng đúng khuôn ports & adapters đã học. Tối thiểu cần:

wallet/ ├── domain/ Wallet, WalletId, Money (VO) ├── application/ │ ├── port/in/ OpenWalletUseCase, DepositUseCase, ListWalletsUseCase │ ├── port/out/ WalletPersistencePort │ └── service/ WalletApplicationService (@Transactional, ownership check) └── adapter/ ├── in/web/ WalletResource (+ DTO: OpenWalletRequest, DepositRequest, WalletResponse) └── out/persistence/ WalletPersistenceAdapter + WalletEntity + WalletPanacheRepo
wallet/adapter/in/web/WalletResource.java — 3 endpoint
@Path(WalletResource.BASE)
@Produces(MediaType.APPLICATION_JSON) @Consumes(MediaType.APPLICATION_JSON)
public class WalletResource {
    public static final String BASE = "/wallets";   # path constant (luật §2)

    private final OpenWalletUseCase openWallet;
    private final DepositUseCase deposit;
    private final ListWalletsUseCase listWallets;
    private final JsonWebToken jwt;
    // constructor injection ...

    @POST @RolesAllowed("USER")
    public RestResponse<WalletResponse> open(@Valid OpenWalletRequest req) {
        var id = UUID.fromString(jwt.getSubject());
        return RestResponse.status(Status.CREATED, openWallet.open(id, req.currencyCode(), req.label()));
    }

    @POST @Path("/{walletId}/deposits") @RolesAllowed("USER")
    public WalletResponse deposit(@PathParam("walletId") UUID walletId,
            @HeaderParam("Idempotency-Key") @NotNull UUID key,   # chỉ NHẬN, chưa enforce (NFR3 ngoài scope)
            @Valid DepositRequest req) {
        var account = UUID.fromString(jwt.getSubject());
        return deposit.deposit(account, walletId, req.amount());
    }

    @GET @RolesAllowed("USER")
    public List<WalletResponse> list() {
        return listWallets.byOwner(UUID.fromString(jwt.getSubject()));
    }
}
Ownership ở service

Trong DepositUseCase.deposit: nạp tiền chỉ khi wallet.accountId == account lấy từ token; nếu không → ném AuthForbiddenException("auth.forbidden") (security.md §3). Validation amount > 0 bằng @DecimalMin(value="0", inclusive=false) trên DTO.

shared/health/DatabaseReadinessCheck.java
@Readiness
@ApplicationScoped
public class DatabaseReadinessCheck implements HealthCheck {
    private final AgroalDataSource ds;
    public DatabaseReadinessCheck(AgroalDataSource ds) { this.ds = ds; }

    @Override public HealthCheckResponse call() {
        try (var c = ds.getConnection()) {
            return HealthCheckResponse.up("database");
        } catch (Exception e) {
            return HealthCheckResponse.down("database");
        }
    }
}

Thực hành · 6/7Circuit Breaker — demo NFR8 (≈ 45 phút)

Mô phỏng một dịch vụ ngoài hay chết (như API tỉ giá FX), rồi bọc fault-tolerance để thấy breaker bảo vệ.

wallet/adapter/out/fx/FlakyFxRateAdapter.java
@ApplicationScoped
public class FlakyFxRateAdapter implements FxRatePort {

    private final Random rnd = new Random();

    @Override
    @Retry(maxRetries = 2, delay = 200)                # thử lại lỗi thoáng qua
    @Timeout(value = 500)                               # quá 500ms coi là lỗi
    @CircuitBreaker(requestVolumeThreshold = 4,         # xét trên 4 lần gọi gần nhất
                    failureRatio = 0.5,                  # >50% lỗi → OPEN
                    delay = 5000)                        # mở 5s rồi half-open
    @Fallback(fallbackMethod = "defaultRate")        # khi lỗi/đang OPEN
    public BigDecimal rateFor(String from, String to) {
        if (rnd.nextBoolean()) throw new RuntimeException("FX provider down");
        return new BigDecimal("1.10");
    }
    // chữ ký fallback PHẢI khớp tham số + kiểu trả về
    public BigDecimal defaultRate(String from, String to) { return BigDecimal.ONE; }
}

Gọi rateFor liên tục (qua một endpoint test tạm hoặc unit test) ~10 lần: ban đầu lúc được lúc fallback, sau khi tỉ lệ lỗi vượt ngưỡng → breaker OPEN, mọi cuộc gọi trả thẳng fallback không chạy method gốc nữa, 5s sau thử lại (half-open).

Đọc thông số

requestVolumeThreshold = cửa sổ xét; failureRatio = ngưỡng lỗi để mở; delay = thời gian OPEN trước khi half-open. Tinh chỉnh và quan sát hành vi đổi.
