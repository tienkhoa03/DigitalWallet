Khi app gọi service ngoài (DB, LLM, FX API), thứ ngoài đó sẽ chậm hoặc chết. 5 pattern dưới đây giúp lỗi cục bộ không kéo sập cả hệ thống.

lỗi vượt ngưỡng hết thời gian chờ CLOSED ───────────────────► OPEN ───────────────────► HALF-OPEN ▲ (cho gọi bình thường) (chặn ngay, (cho gọi thử vài request) │ đếm tỉ lệ lỗi) trả fallback) │ └───────────────────────────────────────────────────────┘ thử thành công → đóng lại (CLOSED)

Phần A · Lý thuyết 3/4Observability — health & metrics

Health: liveness vs readiness

  • Liveness = “app còn sống không?” Nếu fail → orchestrator restart container.
  • Readiness = “app sẵn sàng nhận traffic chưa?” (vd DB đã kết nối). Nếu fail → tạm ngừng route traffic, không restart.
  • Endpoint Quarkus: /q/health/live, /q/health/ready, /q/health.

Metrics: RED method

  • Rate — số request/giây.
  • Errors — tỉ lệ lỗi.
  • Duration — độ trễ (timer/histogram).
  • Repo dùng Micrometer + Prometheus (đã có extension). Prometheus kéo số liệu từ /q/metrics (pull-based).

@Path(WalletResource.BASE)
@Produces(MediaType.APPLICATION_JSON) @Consumes(MediaType.APPLICATION_JSON)
public class WalletResource {
    public static final String BASE = "/wallets";   # path constant (luật §2)

    private final OpenWalletUseCase openWallet;
    private final DepositUseCase deposit;
    private final ListWalletsUseCase listWallets;
    private final JsonWebToken jwt;
    // constructor injection ...

    @POST @RolesAllowed("USER")
    public RestResponse<WalletResponse> open(@Valid OpenWalletRequest req) {
        var id = UUID.fromString(jwt.getSubject());
        return RestResponse.status(Status.CREATED, openWallet.open(id, req.currencyCode(), req.label()));
    }

    @POST @Path("/{walletId}/deposits") @RolesAllowed("USER")
    public WalletResponse deposit(@PathParam("walletId") UUID walletId,
            @HeaderParam("Idempotency-Key") @NotNull UUID key,   # chỉ NHẬN, chưa enforce (NFR3 ngoài scope)
            @Valid DepositRequest req) {
        var account = UUID.fromString(jwt.getSubject());
        return deposit.deposit(account, walletId, req.amount());
    }

    @GET @RolesAllowed("USER")
    public List<WalletResponse> list() {
        return listWallets.byOwner(UUID.fromString(jwt.getSubject()));
    }
}

Phần A · Thực hành 6/7Circuit Breaker — demo NFR8 (≈ 45 phút)

Mô phỏng một dịch vụ ngoài hay chết (như API tỉ giá FX), rồi bọc fault-tolerance để thấy breaker bảo vệ.

wallet/adapter/out/fx/FlakyFxRateAdapter.java
@ApplicationScoped
public class FlakyFxRateAdapter implements FxRatePort {

    private final Random rnd = new Random();

    @Override
    @Retry(maxRetries = 2, delay = 200)                # thử lại lỗi thoáng qua
    @Timeout(value = 500)                               # quá 500ms coi là lỗi
    @CircuitBreaker(requestVolumeThreshold = 4,         # xét trên 4 lần gọi gần nhất
                    failureRatio = 0.5,                  # >50% lỗi → OPEN
                    delay = 5000)                        # mở 5s rồi half-open
    @Fallback(fallbackMethod = "defaultRate")        # khi lỗi/đang OPEN
    public BigDecimal rateFor(String from, String to) {
        if (rnd.nextBoolean()) throw new RuntimeException("FX provider down");
        return new BigDecimal("1.10");
    }
    // chữ ký fallback PHẢI khớp tham số + kiểu trả về
    public BigDecimal defaultRate(String from, String to) { return BigDecimal.ONE; }
}

Gọi rateFor liên tục (qua một endpoint test tạm hoặc unit test) ~10 lần: ban đầu lúc được lúc fallback, sau khi tỉ lệ lỗi vượt ngưỡng → breaker OPEN, mọi cuộc gọi trả thẳng fallback không chạy method gốc nữa, 5s sau thử lại (half-open).

Đọc thông số

requestVolumeThreshold = cửa sổ xét; failureRatio = ngưỡng lỗi để mở; delay = thời gian OPEN trước khi half-open. Tinh chỉnh và quan sát hành vi đổi.

Phần B · Lý thuyết 3/5OIDC — lớp danh tính trên OAuth2

OAuth2 vốn để uỷ quyền truy cập (authorization), không chuẩn hoá “người dùng là ai”. OpenID Connect (OIDC) thêm lớp identity lên trên:

  • ID token (JWT) — cho client biết người dùng là ai (tên, email, sub). Khác access token.
  • Access token — cho resource server; mang scope/role để cấp quyền API.
  • UserInfo endpoint — lấy thêm thông tin người dùng.
  • Discovery — /.well-known/openid-configuration liệt kê mọi endpoint + URL của JWKS.
Phân biệt 2 token

ID token là bằng chứng người dùng đã đăng nhập (dành cho client); access token là chứng từ để gọi API (dành cho resource server). Backend chỉ quan tâm access token.

Phần B · Lý thuyết 4/5JWT & JWKS — verify token thế nào

JWT gồm 3 phần ngăn bởi dấu chấm: header.payload.signature (Base64URL).

header : {"alg":"RS256","kid":"abc","typ":"JWT"} ← thuật toán + id khoá payload : {"sub":"...","exp":...,"realm_access":{"roles":["USER"]}} ← claims signature: ký bằng PRIVATE key của Keycloak

JWKS (JSON Web Key Set) là tập public key Keycloak công khai tại URL trong discovery. Backend tải JWKS một lần, rồi tự verify chữ ký offline (không phải hỏi Keycloak mỗi request) — nhanh & scale. kid trong header chỉ ra dùng key nào.

Các bước verify (Quarkus OIDC làm tự động): kiểm chữ ký bằng JWKS → kiểm iss (issuer đúng realm) → kiểm exp/nbf (cho phép lệch đồng hồ ≤ 30s) → đọc role từ claim.

Bẫy “alg confusion”: tuyệt đối không chấp nhận alg: none hay đổi RS256→HS256. Allow-list thuật toán cứng (security.md §2). Quarkus OIDC mặc định an toàn việc này.

  2. Cấu hình profile keycloak trong application.properties — chỉ kích hoạt khi chạy với -Dquarkus.profile=keycloak, nên path ES256 mặc định không bị đụng:
    application.properties
    %keycloak.quarkus.oidc.auth-server-url=http://localhost:8081/realms/digitalwallet
%keycloak.quarkus.oidc.client-id=digitalwallet-backend
%keycloak.quarkus.oidc.credentials.secret=${KC_BACKEND_SECRET}   # từ env, KHÔNG hardcode
%keycloak.quarkus.oidc.application-type=service
%keycloak.quarkus.oidc.token.audience=account                    # tuỳ cấu hình realm
%keycloak.quarkus.oidc.roles.role-claim-path=realm_access/roles  # map role từ claim này
  3. Giữ nguyên @RolesAllowed("USER") trên WalletResource (vừa dựng ở Phần A) — giờ role đến từ token Keycloak qua role-claim-path ở trên.
  4. Chạy backend ở profile keycloak: 
    export KC_BACKEND_SECRET=...<client secret từ tab Credentials của client backend>...
./mvnw quarkus:dev -Dquarkus.profile=keycloak
Vì sao dùng profile

Profile keycloak cô lập cấu hình OIDC. Chạy quarkus:dev bình thường → vẫn là path ES256 cũ (ADR 0001 còn nguyên). Đây là cách học mà không phá vỡ cấu hình production của repo.

curl -s -X POST \

Phần B · Thực hành 5/5Test bảo mật 401 / 403 (≈ 40 phút)

Đây là ma trận test bắt buộc của repo (security.md §11): unauthenticated → 401, wrong-role → 403.

WalletResourceSecurityTest (QuarkusTest + RestAssured)
@QuarkusTest
class WalletResourceSecurityTest {

    @Test
    void getWallets_khongToken_tra401() {
        given().when().get("/wallets").then().statusCode(401);
    }

    @Test
    void getWallets_saiRole_tra403() {
        // dùng @TestSecurity hoặc token giả role ADMIN cho endpoint chỉ cho USER
        given().auth().oauth2(tokenWithRole("ADMIN"))
            .when().get("/admin-only").then().statusCode(403);
    }
}

Trong CI, dùng quarkus-test-oidc-server (Keycloak giả) hoặc @TestSecurity để không cần Keycloak thật. Học khái niệm trước; tinh chỉnh sau.
