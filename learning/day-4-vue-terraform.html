Phần A — VueJSFrontend reactive trong 5 giờ

A · Lý thuyết 1 — SPA & mô hình reactivity

Một SPA (Single Page Application) tải 1 lần, sau đó JS cập nhật DOM khi state đổi, không reload trang. Trái tim của Vue là reactivity: bạn khai báo state, Vue tự cập nhật phần giao diện phụ thuộc state đó khi nó thay đổi — bạn không tự sờ vào DOM.

state thay đổi ──► Vue phát hiện (reactive proxy) ──► render lại đúng phần phụ thuộc ──► DOM cập nhật (bạn viết) (framework lo) (framework lo) (người dùng thấy)

A · Lý thuyết 2 — Composition API (<script setup>)

Vue 3 khuyến nghị Composition API: logic gom theo “mối quan tâm” thay vì rải theo options. Các viên gạch:

APILàm gìTương đương React
ref(x)State phản ứng cho giá trị đơn (đọc/ghi qua .value)useState
reactive(obj)State phản ứng cho objectuseState với object
computed(fn)Giá trị dẫn xuất, cache theo phụ thuộcuseMemo
watch(src, fn)Chạy side-effect khi nguồn đổiuseEffect
onMounted(fn)Chạy khi component gắn vào DOMuseEffect(...,[])
Ví dụ minh hoạ — một component đếm
<script setup lang="ts">
import { ref, computed } from 'vue'

const count = ref(0)                       // state
const doubled = computed(() => count.value * 2)   // dẫn xuất
function inc() { count.value++ }            // đổi state → UI tự cập nhật
</script>

<template>
  <button @click="inc">count = {{ count }}, x2 = {{ doubled }}</button>
</template>

Cú pháp template: {{ }} nội suy, @click bắt sự kiện, :prop bind, v-for/v-if lặp/điều kiện.

A · Lý thuyết 3 — Pinia (state) & Vue Router

Pinia store

State chia sẻ nhiều component: state (dữ liệu), getters (dẫn xuất), actions (hàm thay đổi/gọi API). Giống Redux nhưng gọn hơn nhiều, không boilerplate.

Vue Router

Map URL → component; beforeEach guard để chặn route cần đăng nhập (tương tự <RequireAuth> bên React).

Thực hành A · 1/2Scaffold app Vue + đăng nhập Keycloak (≈ 90 phút)

  1. Tạo app tách biệt (KHÔNG đụng frontend/ React): 
    pnpm create vue@latest frontend-vue
# chọn: TypeScript ✔, Router ✔, Pinia ✔ (bỏ qua phần khác cho gọn)
cd frontend-vue && pnpm install && pnpm add keycloak-js
  2. Khởi tạo Keycloak client (public digitalwallet-frontend tạo ở Ngày 4):
    src/auth/keycloak.ts
    import Keycloak from 'keycloak-js'

export const keycloak = new Keycloak({
  url: 'http://localhost:8081',
  realm: 'digitalwallet',
  clientId: 'digitalwallet-frontend',
})
// PKCE bật mặc định với client public; gọi keycloak.init({onLoad:'login-required'})
  3. Pinia auth store — giữ token trong memory (không localStorage — security.md §6):
    src/stores/auth.ts
    import { defineStore } from 'pinia'
import { keycloak } from '@/auth/keycloak'

export const useAuth = defineStore('auth', {
  state: () => ({ token: '' as string, username: '' as string }),
  actions: {
    async login() {
      await keycloak.init({ onLoad: 'login-required', pkceMethod: 'S256' })
      this.token = keycloak.token ?? ''
      this.username = keycloak.tokenParsed?.preferred_username ?? ''
    },
    logout() { this.token = ''; keycloak.logout() },
  },
})
  4. Route guard: trong router, beforeEach nếu chưa có token → gọi login().
✅ Checkpoint: mở app → bị đẩy sang trang đăng nhập Keycloak → nhập alice → quay lại app với token trong store.

Thực hành A · 2/2Màn hình ví: list + mở + nạp (≈ 120 phút)

  1. Wallet store — gọi API kèm Bearer token:
    src/stores/wallet.ts
    import { defineStore } from 'pinia'
import { useAuth } from './auth'

const API = '/api'   # Vite proxy → backend (xem bước 4)

export const useWallet = defineStore('wallet', {
  state: () => ({ wallets: [] as Wallet[], loading: false }),
  actions: {
    async fetchWallets() {
      this.loading = true
      const res = await fetch(`${API}/wallets`, {
        headers: { Authorization: `Bearer ${useAuth().token}` },
      })
      this.wallets = await res.json()
      this.loading = false
    },
  },
})
  2. Trang danh sách ví — hiện label + currency + balance cùng nhau (đừng định danh ví bằng currency); format tiền bằng Intl.NumberFormat, KHÔNG toFixed trên number (mất chính xác numeric(19,4)):
    src/views/WalletList.vue
    <script setup lang="ts">
import { onMounted } from 'vue'
import { useWallet } from '@/stores/wallet'
const store = useWallet()
const fmt = (v: string, c: string) =>
  new Intl.NumberFormat(undefined, { style: 'currency', currency: c }).format(Number(v))
onMounted(() => store.fetchWallets())
</script>

<template>
  <p v-if="store.loading">Đang tải…</p>
  <ul v-else>
    <li v-for="w in store.wallets" :key="w.id">
      {{ w.label }} — {{ w.currencyCode }} — {{ fmt(w.balance, w.currencyCode) }}
    </li>
  </ul>
</template>
  3. Form mở ví & nạp tiền: POST /wallets và POST /wallets/{id}/deposits. Form deposit sinh Idempotency-Key (UUID) lúc mount và dùng lại khi resubmit (đừng sinh mới mỗi lần bấm — sẽ thành giao dịch mới).
  4. Vite proxy để tránh CORS khi dev — trỏ /api sang backend:
    vite.config.ts
    server: {
  port: 5173,
  proxy: { '/api': { target: 'http://localhost:8080', changeOrigin: true,
                     rewrite: p => p.replace(/^\/api/, '') } },
}
✅ Checkpoint A: đăng nhập → thấy ví → mở ví mới → nạp tiền → balance cập nhật, tất cả gọi backend đã bảo vệ bằng Keycloak.

Phần B — TerraformInfrastructure as Code trong 3 giờ

B · Lý thuyết 1 — IaC là gì & vòng đời

Thay vì click chuột/gõ lệnh tạo hạ tầng (dễ quên, khó tái lập), bạn mô tả hạ tầng bằng file. Terraform đọc file, so với thực tế, rồi tạo/sửa/xoá để đạt desired state. Đặc tính quan trọng: declarative (mô tả “muốn gì”, không phải “làm từng bước”) & idempotent (apply nhiều lần vẫn ra một kết quả).

.tf files ──► terraform plan ──► terraform apply ──► hạ tầng thật (desired state) (xem trước thay đổi) (thực thi) │ ▼ terraform.tfstate (Terraform nhớ nó đã tạo gì) terraform destroy ──► xoá sạch mọi thứ đã tạo
LệnhLàm gì
terraform initTải provider, khởi tạo thư mục
terraform planHiện trước những gì sẽ thay đổi (an toàn, không thực thi)
terraform applyThực thi để đạt desired state
terraform destroyXoá toàn bộ tài nguyên đã quản lý
State file

terraform.tfstate là “bộ nhớ” của Terraform về tài nguyên đã tạo. Đừng commit nếu chứa secret; với team thật sẽ để remote state (S3…). Học local thì để mặc định.

B · Lý thuyết 2 — HCL & các khối xây dựng

KhốiÝ nghĩa
providerPlugin nói chuyện với 1 nền tảng (docker, aws…)
resourceMột thứ cần tồn tại (container, network, image)
variableTham số đầu vào (port, password) — truyền qua TF_VAR_*
outputGiá trị xuất ra sau apply (tên container, cổng)
dataĐọc thông tin có sẵn (vd pull image)
moduleGói tái sử dụng nhiều resource

Thực hành B · 1/2Viết Terraform dựng hạ tầng local (≈ 105 phút)

Tạo infra/terraform/ tái lập đúng stack đang chạy trong backend/docker-compose.yml, nhưng bằng resource Terraform với Docker provider.

infra/terraform/main.tf (rút gọn — Postgres + network; thêm Redis/Kafka/Keycloak tương tự)
terraform {
  required_providers {
    docker = { source = "kreuzwerker/docker", version = "~> 3.0" }
  }
}
provider "docker" {}

resource "docker_network" "dw" { name = "dw-net" }

resource "docker_image" "postgres" { name = "postgres:16" }

resource "docker_container" "postgres" {
  name  = "dw-postgres"
  image = docker_image.postgres.image_id
  env = [
    "POSTGRES_DB=digitalwallet",
    "POSTGRES_USER=dw",
    "POSTGRES_PASSWORD=${var.db_password}",   # từ biến, KHÔNG hardcode
  ]
  ports { internal = 5432, external = var.db_port }
  networks_advanced { name = docker_network.dw.name }
}
infra/terraform/variables.tf & outputs.tf
# variables.tf
variable "db_password" { type = string, sensitive = true }   # truyền qua TF_VAR_db_password
variable "db_port"     { type = number, default = 5432 }

# outputs.tf
output "postgres_container" { value = docker_container.postgres.name }
output "postgres_port"      { value = var.db_port }
  1. Thêm tiếp resource cho Redis 7, Kafka (KRaft), Keycloak (mount realm-export.json đã export ở Ngày 4) — cùng network dw-net.
  2. Truyền secret qua env, không ghi vào file: 
    export TF_VAR_db_password=...<mật khẩu>...
⚠ Lệch §4.6

Repo mandate Docker Compose. Terraform sống dưới infra/terraform/ như spike song song, không thay compose của team trừ khi có ADR.

Thực hành B · 2/2apply / destroy + Demo End-to-End (≈ 60 phút)

  1. Khởi tạo & xem trước: 
    cd infra/terraform
terraform init       # tải Docker provider
terraform plan       # đọc kỹ: "X to add" — chưa tạo gì cả
  2. Thực thi: 
    terraform apply      # gõ yes → các container lên
docker ps            # thấy dw-postgres, dw-redis, dw-kafka, dw-keycloak
terraform output     # xem tên container + cổng
  3. Chạy demo E2E:
    • Backend: cd backend && ./mvnw quarkus:dev -Dquarkus.profile=keycloak
    • Vue: pnpm --dir frontend-vue dev → mở http://localhost:5173
    • Đăng nhập Keycloak → mở ví → nạp tiền → balance cập nhật.
  4. Dọn dẹp để thấy trọn vòng đời IaC: 
    terraform destroy    # gõ yes → mọi container/network biến mất sạch
✅ Checkpoint B: một lệnh apply dựng cả hạ tầng, một lệnh destroy xoá sạch. Demo E2E chạy thông từ Vue → Keycloak → backend → DB.
