DIP = nền của Hexagonal

Bình thường: nghiệp vụ → gọi → DB (nghiệp vụ phụ thuộc DB). DIP đảo ngược: nghiệp vụ định nghĩa interface (port), DB đi implement (adapter). Giờ mũi tên phụ thuộc trỏ vào nghiệp vụ. Đó chính xác là cái hexagonal làm ở quy mô module.

KHÔNG có DIP: Service ──phụ thuộc──► JwtIssuer (chi tiết) CÓ DIP: Service ──phụ thuộc──► «interface» TokenIssuerPort ◄──implements── JwtIssuer (trừu tượng do domain sở hữu)

Lý thuyết · 2/4DDD tactical — từ vựng để mô hình hoá nghiệp vụ

Entity vs Value Object — khác biệt cốt lõi

Entity

Có danh tính (id) tồn tại theo thời gian. Hai entity bằng nhau khi cùng id, dù thuộc tính khác. Ví dụ: Account — đổi email vẫn là account đó.

Value Object

Không có id; định danh bằng giá trị. Hai VO bằng nhau khi mọi field bằng nhau, và nên bất biến. Ví dụ: Email, BaseCurrency, MoneyAmount. Dùng VO để diệt “primitive obsession” (đừng để email là String trần).

Aggregate & invariant

Aggregate là cụm object được xem như một khối nhất quán, có một “gốc” (aggregate root) là cửa vào duy nhất. Mọi thay đổi đi qua root để root giữ invariant (luật luôn đúng). Ví dụ tương lai: Wallet là root, balance không bao giờ âm — luật đó được bảo vệ bên trong Wallet, không rải rác ở service.

Lý thuyết · 3/4Layered → Hexagonal (Ports & Adapters)

Vấn đề của “layered” thuần

Kiểu layered (api → service → persistence) rất phổ biến, nhưng nếu service import thẳng class persistence/Hibernate thì nghiệp vụ bị dính vào DB: muốn test phải có DB, muốn đổi DB phải sửa nghiệp vụ. Hexagonal sửa điều đó bằng DIP.

Panache khuyến khích entity “thông minh” (Active Record). Hexagonal “tinh khiết” lại muốn domain tách rời entity → phải viết 2 hàm map (toDomain/toEntity). Lợi: test domain không cần DB, đổi ORM chỉ sửa adapter. Hại: thêm boilerplate. Hãy tự ghi lại cảm nhận sau khi làm — đó là bài học thật về trade-off, không có đáp án “đúng tuyệt đối”.

Thực hành · 1/6Vẽ port map trước khi gõ code (≈ 25 phút)

Đừng vội code. Liệt kê ra giấy: app account làm được gì (→ in-ports) và cần gì từ ngoài (→ out-ports).

In-ports (use cases)

  • RegisterAccountUseCase.register(cmd) → AccountId
  • AuthenticateUseCase.login(cmd) → AccessToken
  • GetProfileUseCase.byId(id) → Profile (từ Ngày 1)

Out-ports (cần từ ngoài)

  • LoadAccountPort.findByEmail / findById
  • SaveAccountPort.save(Account)
  • PasswordHasherPort.hash / verify
  • TokenIssuerPort.issue(account)

    AccountId register(RegisterAccountCommand command);
    // command là record thuần trong application/, KHÁC với DTO web
}
Command vs DTO

DTO (CreateAccountRequest) thuộc web, mang validation Bean. Command thuộc application, là input “sạch” của use case. Adapter web dịch DTO → command. Tách ra để domain không phụ thuộc annotation web.

Thực hành · 4/6Application service (di chuyển logic) (≈ 60 phút)

Bê logic từ AccountService/AuthService cũ vào đây, nhưng chỉ gọi out-port (interface), không gọi Panache/JwtIssuer trực tiếp.

application/service/AccountApplicationService.java
@ApplicationScoped
public class AccountApplicationService
        implements RegisterAccountUseCase, AuthenticateUseCase, GetProfileUseCase {

    private final LoadAccountPort loadPort;
    private final SaveAccountPort savePort;
    private final PasswordHasherPort hasher;
    private final TokenIssuerPort tokenIssuer;

    public AccountApplicationService(LoadAccountPort loadPort, SaveAccountPort savePort,
                                   PasswordHasherPort hasher, TokenIssuerPort tokenIssuer) {
        this.loadPort = loadPort; this.savePort = savePort;
        this.hasher = hasher;     this.tokenIssuer = tokenIssuer;
    }

    @Override @Transactional
    public AccountId register(RegisterAccountCommand cmd) {
        if (loadPort.findByEmail(cmd.email()).isPresent())
            throw new ConflictException("account.duplicate_email", "Email exists");
        var account = new Account(AccountId.newId(), cmd.email(),
                hasher.hash(cmd.rawPassword()), cmd.baseCurrency(), AccountRole.USER);
        return savePort.save(account).id();
    }
}
✅ Checkpoint: file này không import AccountEntity, AccountRepository, JwtIssuer. Chỉ import port (interface) + domain.

5.1 — Persistence adapter (map entity ↔ domain)

adapter/out/persistence/AccountPersistenceAdapter.java
@ApplicationScoped
public class AccountPersistenceAdapter implements SaveAccountPort, LoadAccountPort {

    private final AccountPanacheRepository repo;   # Panache repo nằm trong adapter này
    public AccountPersistenceAdapter(AccountPanacheRepository repo) { this.repo = repo; }

    @Override public Account save(Account a) {
        var e = toEntity(a);
        repo.persist(e);
        return toDomain(e);
    }
    @Override public Optional<Account> findByEmail(Email email) {
        return repo.find("email", email.value()).firstResultOptional().map(this::toDomain);
    }

    private Account toDomain(AccountEntity e) { /* ... map fields ... */ }
    private AccountEntity toEntity(Account a) { /* ... map fields ... */ }
}

Thực hành · 6/6 (tuỳ chọn)Chốt “dependency rule” bằng ArchUnit (≈ 30 phút)

ArchUnit là thư viện test kiến trúc: viết “luật” và CI sẽ fail nếu ai đó phá luật.

src/test/java/.../architecture/HexagonalArchTest.java
@AnalyzeClasses(packages = "com.digitalwallet.account")
class HexagonalArchTest {
    @ArchTest
    static final ArchRule domain_khong_phu_thuoc_ha_tang =
        noClasses().that().resideInAPackage("..account.domain..")
          .should().dependOnClassesThat()
          .resideInAnyPackage("jakarta.persistence..", "io.quarkus..", "..adapter..");
}
