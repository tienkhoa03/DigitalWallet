Hôm nay bạn để một Identity Provider (Keycloak) lo việc đăng nhập & phát token, còn backend chỉ verify token và map role. Lý thuyết · 3/5OIDC — lớp danh tính trên OAuth2

OAuth2 vốn để uỷ quyền truy cập (authorization), không chuẩn hoá “người dùng là ai”. OpenID Connect (OIDC) thêm lớp identity lên trên:

  • ID token (JWT) — cho client biết người dùng là ai (tên, email, sub). Khác access token.
  • Access token — cho resource server; mang scope/role để cấp quyền API.
  • UserInfo endpoint — lấy thêm thông tin người dùng.
  • Discovery — /.well-known/openid-configuration liệt kê mọi endpoint + URL của JWKS.
Phân biệt 2 token

ID token = “bằng chứng đăng nhập” cho app. Access token = “vé vào cửa” API. Backend chỉ quan tâm access token.

Lý thuyết · 4/5JWT & JWKS — verify token thế nào

JWT gồm 3 phần ngăn bởi dấu chấm: header.payload.signature (Base64URL).

header : {"alg":"RS256","kid":"abc","typ":"JWT"} ← thuật toán + id khoá payload : {"sub":"...","exp":...,"realm_access":{"roles":["USER"]}} ← claims signature: ký bằng PRIVATE key của Keycloak

JWKS (JSON Web Key Set) là tập public key Keycloak công khai tại URL trong discovery. Backend tải JWKS một lần, rồi tự verify chữ ký offline (không phải hỏi Keycloak mỗi request) — nhanh & scale. kid trong header chỉ ra dùng key nào.

Các bước verify (Quarkus OIDC làm tự động): kiểm chữ ký bằng JWKS → kiểm iss (issuer đúng realm) → kiểm exp/nbf (cho phép lệch đồng hồ ≤ 30s) → đọc role từ claim.

Bẫy “alg confusion”: tuyệt đối không chấp nhận alg: none hay đổi RS256→HS256. Allow-list thuật toán cứng (security.md §2). Quarkus OIDC mặc định an toàn việc này.

Lý thuyết · 5/5Mô hình Keycloak + Quarkus OIDC

ClientMột ứng dụng. public (SPA, không giữ secret) / confidential (server, có secret) / bearer-only
Rolerealm role (toàn realm) hoặc client role (riêng 1 client)
User / GroupNgười dùng; group gom role cho tiện gán

Quarkus OIDC có 2 kiểu app: service (chỉ verify bearer token — backend của ta) và web-app (tự lo redirect đăng nhập — cho app server-side render). Ta dùng service.

Thực hành · 1/5Dựng Keycloak bằng Docker (≈ 30 phút)

Tạm thêm service vào backend/docker-compose.yml (Ngày 5 sẽ chuyển sang Terraform). Dùng dev mode cho nhanh.

thêm vào backend/docker-compose.yml (services:)
  keycloak:
    image: quay.io/keycloak/keycloak:26.0
    command: ["start-dev", "--import-realm"]
    environment:
      KEYCLOAK_ADMIN: admin
      KEYCLOAK_ADMIN_PASSWORD: admin
    ports: ["8081:8080"]
    volumes:
      - ./infra/keycloak:/opt/keycloak/data/import   # để import realm-export.json
    networks: [dw-net]
Bật & mở console
docker compose -f backend/docker-compose.yml up -d keycloak
# Mở http://localhost:8081  → đăng nhập admin/admin

Thực hành · 2/5Tạo realm, client, role, user (≈ 45 phút)

  1. Tạo realm: góc trên trái → Create realm → tên digitalwallet → Create.
  2. Tạo realm roles: menu Realm roles → Create role → USER; lặp lại với ADMIN.
  3. Client backend: Clients → Create → Client ID digitalwallet-backend, type OpenID Connect. Bước Capability: bật Client authentication (confidential). Lưu.
  4. Client frontend (cho Vue): tạo digitalwallet-frontend, tắt Client authentication (public), bật Standard flow. Valid redirect URIs: http://localhost:5173/*; Web origins: http://localhost:5173.
  5. Tạo user: Users → Create user → username alice → tab Credentials đặt password (tắt Temporary) → tab Role mapping → Assign role USER.
  6. Export realm để tái lập: docker exec ... /opt/keycloak/bin/kc.sh export hoặc dùng UI Realm settings → Action → Partial export → lưu thành backend/infra/keycloak/realm-export.json.
✅ Checkpoint: mở http://localhost:8081/realms/digitalwallet/.well-known/openid-configuration thấy JSON liệt kê token_endpoint, jwks_uri…

Thực hành · 3/5Tích hợp Quarkus OIDC (theo profile) (≈ 60 phút)

  1. Thêm extension: 
    cd backend
./mvnw quarkus:add-extension -Dextensions="oidc"
  2. Cấu hình profile keycloak trong application.properties — chỉ kích hoạt khi chạy với -Dquarkus.profile=keycloak, nên path ES256 mặc định không bị đụng:
    application.properties
    %keycloak.quarkus.oidc.auth-server-url=http://localhost:8081/realms/digitalwallet
%keycloak.quarkus.oidc.client-id=digitalwallet-backend
%keycloak.quarkus.oidc.credentials.secret=${KC_BACKEND_SECRET}   # từ env, KHÔNG hardcode
%keycloak.quarkus.oidc.application-type=service
%keycloak.quarkus.oidc.token.audience=account                    # tuỳ cấu hình realm
%keycloak.quarkus.oidc.roles.role-claim-path=realm_access/roles  # map role từ claim này
  3. Giữ nguyên @RolesAllowed("USER") trên WalletResource — giờ role đến từ token Keycloak qua role-claim-path ở trên.
  4. Chạy backend ở profile keycloak: 
    export KC_BACKEND_SECRET=...<client secret từ tab Credentials của client backend>...
./mvnw quarkus:dev -Dquarkus.profile=keycloak
Vì sao dùng profile

Profile keycloak cô lập cấu hình OIDC. Chạy quarkus:dev bình thường → vẫn là path ES256 cũ (ADR 0001 còn nguyên). Đây là cách “học mà không phá”.

Thực hành · 4/5Lấy token & gọi API (≈ 45 phút)

Lấy access token (password grant — chỉ để test)
curl -s -X POST \
  http://localhost:8081/realms/digitalwallet/protocol/openid-connect/token \
  -d grant_type=password \
  -d client_id=digitalwallet-frontend \
  -d username=alice -d password=<pw> | jq -r .access_token
  1. Dán token vào jwt.io để xem payload: tìm realm_access.roles có USER, iss đúng realm, exp còn hạn.
  2. Gọi API đã bảo vệ: 
    TOKEN=...<access_token>...
curl -i http://localhost:8080/wallets -H "Authorization: Bearer $TOKEN"   # 200
✅ Checkpoint: có token Keycloak → GET /wallets trả 200. Đây là lần đầu API của bạn được bảo vệ bởi một IdP thực thụ.

Thực hành · 5/5Test bảo mật 401 / 403 (≈ 40 phút)

Đây là ma trận test bắt buộc của repo (security.md §11): unauthenticated → 401, wrong-role → 403.

WalletResourceSecurityTest (QuarkusTest + RestAssured)
@QuarkusTest
class WalletResourceSecurityTest {

    @Test
    void getWallets_khongToken_tra401() {
        given().when().get("/wallets").then().statusCode(401);
    }

    @Test
    void getWallets_saiRole_tra403() {
        // dùng @TestSecurity hoặc token giả role ADMIN cho endpoint chỉ cho USER
        given().auth().oauth2(tokenWithRole("ADMIN"))
            .when().get("/admin-only").then().statusCode(403);
    }
}

Trong CI, dùng quarkus-test-oidc-server (Keycloak giả) hoặc @TestSecurity để không cần Keycloak thật. Học khái niệm trước; tinh chỉnh sau.
