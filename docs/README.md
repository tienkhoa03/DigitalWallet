# docs/

This folder is the human-readable source of truth for the DigitalWallet repository: it describes **what** the system is and **why** it is shaped the way it is.

Implementation-level **how** (coding rules, lint configuration, prompt scaffolds) lives under [../.claude/rules/](../.claude/rules/) `(not yet present — generated in step 3)`. The mandates that govern both folders live in [../CLAUDE.md](../CLAUDE.md), and the authoritative product input is [../project-info.md](../project-info.md).

## Sub-folders

| Folder | Purpose |
|---|---|
| [onboarding/](onboarding/README.md) | Bring up the local stack from a fresh clone. |
| [architecture/](architecture/README.md) | High-level system shape, module layout, deployment topology, config matrix. |
| [api/](api/README.md) | REST endpoint catalog inferred from the functional requirements. |
| [database/](database/README.md) | Entity model, table descriptions, and naming conventions. |
| [database/migrations.md](database/migrations.md) | Migration tooling, naming, forward-only policy, seed-data handling. |
| [domain-knowledge/](domain-knowledge/README.md) | Product framing, glossary, and persona-level user journeys. |
| [business-rules/](business-rules/README.md) | NFR enforcement matrix and per-epic functional rule pages. |
| [testing/](testing/README.md) | Test framework choices, coverage floors, and discipline rules. |
| [decisions/](decisions/README.md) | Architecture Decision Records (ADRs) — template plus one file per decision in [../project-info.md §10](../project-info.md#10-open-architectural-decisions-adrs-to-write). |
| [plans/](plans/) | Implementation plans generated by `/make-plan` from step 5 onwards. |

## How to use this folder

- **Onboarding a new contributor?** Start at [onboarding/](onboarding/README.md), then read [architecture/](architecture/README.md) and [domain-knowledge/](domain-knowledge/README.md).
- **Adding a feature?** Open the relevant epic page under [business-rules/](business-rules/README.md), then the matching endpoint row in [api/](api/README.md) and table description in [database/](database/README.md).
- **Making an architectural change?** Draft a new ADR using [decisions/template.md](decisions/template.md), link it from the pull request, and update the affected pages in [architecture/](architecture/README.md) once accepted.
- **Changing a non-functional invariant?** Update the matrix in [business-rules/README.md](business-rules/README.md) so the NFR-to-layer mapping stays current.
- **Drafting an implementation plan?** Land it under [plans/](plans/) — the folder is intentionally empty until `/make-plan` produces output.

## Conventions

- Pages avoid first-person voice ("we", "I").
- Claims that depend on code not yet written are suffixed `(verify)` or labelled `(spec — not yet implemented)`.
- Cross-links use relative paths.
- Detailed coding conventions live outside this folder under [../.claude/rules/](../.claude/rules/) `(not yet present)`.
