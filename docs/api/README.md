# API

This page catalogs the REST endpoint surface inferred from the functional requirements in [../../project-info.md §5](../../project-info.md#5-functional-requirements-epics--frs). All endpoints are `(spec — not yet implemented)` — paths and payload shapes are illustrative until backed by code.

## Conventions

- **Style:** REST over HTTPS, JSON wire format.
- **API library:** JAX-RS (RESTEasy Reactive) under Quarkus 3.x ([../../project-info.md §4.1](../../project-info.md#41-backend)).
- **Documentation:** OpenAPI generated by the Quarkus SmallRye OpenAPI extension; hand-maintained at the resource layer ([../../project-info.md §13](../../project-info.md#13-coding-conventions-highest-level-project-wide)).
- **Auth:** all endpoints except sign-up and login require a stateless JWT (`Authorization: Bearer <token>`), signed with ES256. RBAC is enforced at the service layer, not only in the controller ([../../project-info.md §8](../../project-info.md#8-security-baseline)).
- **Idempotency:** every mutating money endpoint requires an `Idempotency-Key: <uuid>` HTTP header; replays return the original outcome (NFR3). See [../business-rules/README.md](../business-rules/README.md).
- **Rate limiting (Redis token bucket):** `POST /transfers` 10/min/user, `POST /advisor/*` 5/hour/user ([../../project-info.md §8](../../project-info.md#8-security-baseline)).
- **Currencies and timestamps:** ISO 4217 currency codes; UTC ISO-8601 timestamps ([../../project-info.md §13](../../project-info.md#13-coding-conventions-highest-level-project-wide)).
- **Real-time:** the WebSocket surface is documented under [../architecture/README.md](../architecture/README.md). It carries the fraud-alert stream (FR3.2), budget-threshold alerts (FR5.1), and the asynchronous LLM advisor reply (NFR8).

### Epic 2 — Fraud Detection Engine ([rules](../business-rules/fraud-detection-engine-rules.md))

FR2.1 (velocity) and FR2.2 (volume) run **synchronously** on the money path before the wallet lock (NFR9) and reject offending requests inline with HTTP 422 `fraud.velocity_exceeded` / `fraud.volume_exceeded`; FR2.4 (account suspension) rejects suspended accounts with HTTP 403 `account.suspended`. Cross-event analysis, the suspension-policy decision (FR2.4), and analyst alerting (FR2.5) run in Kafka-consumer threads. FR2.3 (event publishing) is satisfied by the outbox poller for both successful and blocked attempts. Tunable thresholds are environment-driven ([../architecture/README.md#7-config--profiles](../architecture/README.md#7-config--profiles)); a fraud-analyst tuning endpoint may be added in a later iteration `(verify)`.

|---|---|---|---|---|---|---|
| `GET` | `/fraud/alerts` | List recent fraud alerts (audit/inspection). | `ADMIN` / `FRAUD_ANALYST` | query: `from`, `to`, `account_id?` | `[ { alert_id, account_id, rule, evidence, raised_at } ]` | `auth.forbidden` |

## Error response shape

All endpoints surface errors through a single JAX-RS exception mapper that wraps the project's `DomainException` hierarchy ([../../project-info.md §13](../../project-info.md#13-coding-conventions-highest-level-project-wide)). The canonical envelope is:

```json
{
  "error_key": "<dot.separated.identifier>",
  "message": "<human-readable description>"
}
```

- `error_key` is the stable contract; clients branch on it.
- `message` is informational and may be localised; clients must not branch on it.
- HTTP status mapping is set per `error_key` at the mapper.
- RFC 9457 (Problem Details for HTTP APIs) is listed as a candidate for the error envelope in [../../project-info.md §15](../../project-info.md#15-reference-materials) — if adopted, the envelope above maps onto `type` / `title` / `detail` fields `(verify when ADR lands)`.
- Validation failures from Hibernate Validator are normalised onto the same envelope with `error_key: "validation.invalid_payload"` and a `details` array of field-level violations `(verify)`.
