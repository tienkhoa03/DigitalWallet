0 · Giới thiệu & mục tiêu học

DigitalWallet là nền tảng ví nội bộ đa tiền tệ. MVP chỉ làm Epic 1 — Core Wallet Management (FR1.1–FR1.4): đăng ký/đăng nhập → mở nhiều ví theo từng loại tiền → nạp / rút / chuyển (idempotency + khoá đồng thời + rate limit) → xem sao kê. Fraud / Dashboard / PFM / AI Advisor hoãn (kiến trúc đã chừa chỗ: cột fraud_status, bảng outbox_event vẫn ship).

Backend / Kiến trúc

  • Modular monolith + vertical slice
  • Transactional Outbox (atomic ledger+event)
  • Hybrid concurrency: Redis lock → DB PESSIMISTIC_WRITE
  • Idempotency state machine (IN_FLIGHT/COMPLETED)
  • JWT ES256, Argon2id, RBAC 2 tầng
  • Flyway forward-only, Testcontainers, JaCoCo gate
  • BigDecimal/numeric(19,4), FX snapshot

Frontend / Chất lượng

  • Vue 3 + TS strict + Pinia + TanStack Query (Vue Query)
  • VeeValidate + Zod mirror validation backend
  • Tiền là decimal string — không mất độ chính xác
  • Route guard, error boundary, XSS-safe
  • Shared HTTP client: inject JWT + 401→logout + map error
  • Vitest/@testing-library/vue + Playwright golden path

3 · Kiến trúc 2 luồng

① Luồng đồng bộ (money path)

Trên chính HTTP request thread:

  1. REST endpoint nhận request
  2. (Fraud pre-check — hoãn MVP)
  3. Idempotency check (claim row IN_FLIGHT)
  4. Redis lock theo wallet_id (fail nhanh)
  5. Mở @Transactional
  6. SELECT … FOR UPDATE (PESSIMISTIC_WRITE)
  7. Ghi ledger row + outbox row trong 1 tx
  8. Commit → release lock → trả response

② Luồng bất đồng bộ (Kafka)

Thread pool riêng — hoãn MVP:

  • Outbox poller → topic transaction-events
  • Fraud / PFM / Dashboard / Advisor consume
  • Fan-out realtime qua WebSocket

MVP chỉ ghi outbox; chưa poller/consumer nào đọc → bảng outbox_event tích lại (chấp nhận được, là điểm audit).

💡 Vì sao tách 2 luồng + Outbox?
Nếu HTTP handler tự gọi Kafka → "dual write": DB commit OK nhưng publish fail = lệch dữ liệu. Transactional Outbox: ghi event vào cùng transaction DB với ledger; poller riêng mới publish. Request trả nhanh (NFR5) + không mất event (NFR2). Consumer phải idempotent (at-least-once). Pattern kinh điển event-driven.

Cấu trúc module (per-module hexagonal — ports & adapters)

backend/src/main/java/com/digitalwallet/
├── shared/        # money, idempotency, outbox, lock, ratelimit, security, exception, validation
├── account/       # FR1.1 — signup, login, role, base_currency (1 hexagon)
│   ├── domain/        #   model thuần (KHÔNG annotation JPA) + rule nghiệp vụ
│   ├── application/   #   port/in (use case) · port/out (SPI) · service (use-case impl, @Transactional + RBAC)

⚠️ Luật phụ thuộc & import (bị code-review chặn)
  • Dependency chỉ hướng vào trong: adapter → application → domain; framework chỉ nằm trong adapter/. domain/ + application/ không có annotation JPA/JAX-RS.
  • adapter/ của module A KHÔNG import domain//application//adapter/ của module B.
  • Application service phụ thuộc outbound port (application/port/out), không phụ thuộc adapter cụ thể.
  • pfm/ KHÔNG có outbound persistence adapter trên transaction/wallet/outbox_event (NFR6).
  • Giao tiếp cross-module: qua Kafka topic hoặc port trong shared/.

🔐 Tham chiếu · Money path chi tiết (NFR1+2+3)

Đây là "trái tim" dự án — Phase 3 dựng, Phase 4/5 tái dùng. Thứ tự cố định (sai thứ tự = bug đồng thời / deadlock):

POST /wallets/{walletId}/deposits   (Idempotency-Key: <uuidv7>)
 └─ 1. Idempotency: INSERT idempotency_record (user,endpoint,key) ON CONFLICT
 │       ├─ đã COMPLETED + cùng request_hash → trả response_snapshot (replay)   [NFR3]
 │       ├─ đã COMPLETED + khác request_hash → 409 idempotency.replay_conflict
 │       └─ đang IN_FLIGHT → 409 (retry sau)
 └─ 2. Redis lock key=wallet:{walletId}, TTL ngắn (fail-fast)                   [NFR1 outer]
 │       └─ không lấy được → 409 wallet.locked
 └─ 3. @Transactional BEGIN
 │       4. SELECT … FOR UPDATE wallet  (LockModeType.PESSIMISTIC_WRITE)        [NFR1 inner]
 │       5. validate (số dư đủ cho withdraw? currency khớp?)
 │       6. wallet.balance += amount  (BigDecimal)
 │       7. INSERT transaction (ledger row)
 │       8. INSERT outbox_event (DepositCompleted)  — cùng tx!                  [NFR2]
 │       9. UPDATE idempotency_record SET status=COMPLETED, response_snapshot=…
 │      COMMIT
 └─ 10. finally { Redis lock RELEASE }
 └─ 11. return { transaction_id, balance_after, event_timestamp }
💡 Vì sao 2 lớp khoá?
Redis lock (outer) fail nhanh — chặn spam click/retry rẻ tiền, không đụng DB. DB FOR UPDATE (inner) là authoritative — đảm bảo ACID thật sự ngay cả khi Redis lỗi. Redis không phải nguồn sự thật; trạng thái phải dựng lại được từ Postgres. Transfer (Phase 5): khoá 2 ví theo thứ tự wallet_id UUID tăng dần để A→B và B→A đồng thời không deadlock.

Phase 1 · Signup + Login + JWT (FR1.1) Đang scaffold

Bạn xây gì

shared/exception/ (sealed DomainException + DomainExceptionMapper + ConstraintViolationExceptionMapper + ErrorResponse), shared/security/ (Argon2Hasher, JwtIssuer, JwtSigningKeyProvider, UserRole), shared/validation/ (@CurrencyCode), module user/ + Flyway V1, endpoint POST /users + POST /auth/login.

Đặc tả cụ thể (chốt trong plan)

  • Argon2id: memory 65536 KB (64 MiB), iterations 3, parallelism 1, salt 16B, hash 32B. Lưu chuỗi tự mô tả $argon2id$v=19$m=65536,t=3,p=1$<salt>$<hash>. So sánh constant-time bằng MessageDigest.isEqual.
  • JWT claims: sub=user_id, groups=["USER"], iss=digitalwallet, aud=digitalwallet-api, exp=iat+3600. Thuật toán ES256 cứng (smallrye.jwt.verify.algorithm=ES256), skew 30s (smallrye.jwt.expiration.grace=30).
  • Email: @Email; uniqueness case-insensitive qua index LOWER(email); lookup findByEmailLower.
  • Enumeration resistance: login email-không-tồn-tại vẫn hash với sentinel rồi ném auth.invalid_credentials giống hệt nhánh sai mật khẩu (cùng message, cùng wall-clock).

Code mẫu (đã có thật trong repo — UserService.signup)

@Transactional
public CreateUserResponse signup(CreateUserRequest request) {
    String emailLower = request.email().toLowerCase(Locale.ROOT);
    userRepository.findByEmailLower(emailLower).ifPresent(existing -> {
        throw new ConflictException("user.email_taken", "Email is already registered");
    });
    UserEntity user = new UserEntity();
    user.id = UUID.randomUUID();
    user.email = request.email();
    user.passwordHash = hasher.hash(request.password());   // Argon2id
    user.role = UserRole.USER;
    user.baseCurrency = request.baseCurrency();
    user.fraudStatus = "ACTIVE";
    user.createdAt = clock.instant();                      // Clock injected (test fix thời gian)
    userRepository.persist(user);
    return new CreateUserResponse(user.id, user.createdAt);
}

💡 Kiến thức (rất giàu)

  • JWT ES256: vì sao từ chối alg:none/HS256 (JWT confusion attack — kẻ tấn công ký HS256 bằng chính public key). Allow-list thuật toán cứng.
  • Argon2id vs SHA-256/MD5: memory-hard chống brute-force GPU.
  • Constant-time + enumeration resistance: không để kẻ tấn công đoán email tồn tại qua thời gian phản hồi hoặc thông điệp khác nhau.
  • Constructor injection (cấm field injection) — dễ test, phụ thuộc rõ ràng.
  • Inject Clock thay vì Instant.now() — test fix được thời gian.

🟣 Cạm bẫy

🟣 Hay sai ở Phase 1
  • Trả password_hash ra DTO response → vi phạm security §7. Response chỉ {user_id,created_at}.
  • Log full email / full JWT / Authorization header → cấm (PII / SOC 2).
  • Quên quarkus.smallrye-jwt.enabled=true → verifier không chạy.
  • Hai nhánh login fail trả message khác nhau → lộ enumeration.

Phase 5 · Transfer two-leg (FR1.3) Khó nhất

Bạn xây gì

POST /transfers: debit + credit trong 1 transaction, chung transfer_id, snapshot FX trên cả 2 leg. Recipient bằng to_user_id; ví nhận chọn server-side theo currency_code (nhiều ví → ví updated gần nhất).

Đặc tả lỗi

  • transfer.recipient_not_found (không tìm thấy ví nhận phù hợp currency) · 422
  • transfer.same_wallet (chuyển vào chính ví nguồn) · 422
  • transfer.fx_rate_missing (thiếu rate cho cặp currency) · 422
  • wallet.insufficient_funds · 422 · wallet.currency_mismatch · 422
  • ratelimit.exceeded · 429 (Phase 6 mới wire, nhưng endpoint đã đánh dấu rate-limited)

💡 Kiến thức

  • Atomic two-leg: 2 transaction row (1 DEBIT, 1 CREDIT) phải cùng commit/rollback — bản chất "chuyển khoản". Một leg fail → rollback toàn bộ.
  • Deadlock avoidance: khoá Redis 2 ví theo thứ tự wallet_id UUID tăng dần. Nếu A→B và B→A đồng thời mà khoá tuỳ tiện → deadlock kinh điển. Kỹ thuật "lock ordering".
  • FX snapshot tại thời điểm giao dịch: chốt rate (from→to), lưu exchange_rate trên cả 2 leg với cùng giá trị (Convention A). Rate không bao giờ dùng để định giá lại balance đã lưu.
🟣 Cạm bẫy
  • Khoá ví theo thứ tự request thay vì thứ tự UUID → deadlock dưới tải.
  • Quên snapshot FX trên leg credit (chỉ ghi leg debit) → audit lệch.
  • Để lộ to_wallet_id ra payload request — sai, người gửi chỉ biết to_user_id.

Test bắt buộc

  • Concurrent A→B và B→A đồng thời → không deadlock, cả hai hoàn tất hoặc retry sạch.
  • Cross-currency: exchange_rate trên 2 leg bằng nhau; tổng tiền bảo toàn theo rate.
